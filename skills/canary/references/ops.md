# Canary Operations

## Architecture

Two Fly.io apps:

| App | Purpose | Deploy from |
|-----|---------|-------------|
| `canary-obs` | Core: error ingestion, health checking, webhooks | repo root |
| `canary-triage` | Companion: receives webhooks, creates GitHub issues via LLM | `triage/` |

SQLite + Litestream replication. Single-writer (pool_size: 1).

## Deploy

```bash
# Core service (from repo root)
flyctl deploy --app canary-obs --remote-only

# Triage service (MUST cd into triage/ — wrong cwd = wrong Dockerfile)
cd triage && flyctl deploy --app canary-triage --remote-only
```

## Bootstrap

First boot logs the bootstrap API key — grep for `"Bootstrap API key:"`.
Store it; it won't be shown again. Use it to create additional keys via
`POST /api/v1/keys`.

## Nuclear Reset

SQLite WAL keeps file handles open. Must stop first:

```bash
flyctl machines stop <id> --app canary-obs
flyctl ssh console --app canary-obs -C "rm -f /data/canary.db /data/canary.db-wal /data/canary.db-shm"
flyctl machines start <id> --app canary-obs
```

## Triage Service

- Webhook route: `/webhooks/canary` (plural)
- Webhook secret: generated BY Canary on registration, must match `CANARY_WEBHOOK_SECRET` in triage env
- GitHub token: needs `issues:write` on target repos
- `SERVICE_REPOS` env maps service→repo: `{"canary-triage":"misty-step/canary"}`
- Circuit breaker: opens after 10 failures, probes every 5 min

## Self-Monitoring

Both services report errors to Canary:
- Core: direct ingest via `Canary.Errors.Ingest.ingest/1` (no HTTP)
- Triage: HTTP to canary-obs via fire-and-forget `Task.start/1`
